# SQL Database Bootstrap and Automation

This document describes how the SQL database is automatically configured as part of the CI/CD deployment process.

## Overview

The Capacity Planning Dashboard uses Azure SQL Database with Azure AD managed identity authentication. To enable database access, the App Service's managed identity must be created as a contained database user with appropriate role grants.

Previously, this required manual steps in the Azure Portal. Now it is **fully automated** as part of the deployment workflow.

## Architecture

### Components

1. **SQL Initialization Script** (`sql/init-managed-identity.sql`)
   - T-SQL script that creates the database user from the managed identity
   - Grants `db_datareader` and `db_datawriter` roles
   - Idempotent - safe to run multiple times

2. **Bootstrap Script** (`scripts/bootstrap-sql-managed-identity.ps1`)
   - PowerShell script executed by GitHub Actions
   - Authenticates to Azure using the AZURE_CREDENTIALS secret
   - Substitutes the App Service name into the SQL script
   - Executes the SQL script against the database using Azure AD authentication

3. **GitHub Actions Workflow** (`.github/workflows/deploy.yml`)
   - Both `deploy` and `deploy-manual` jobs include the SQL bootstrap step
   - Runs after the app is deployed to App Service
   - Restarts the app to load the new database permissions

## How It Works

### Deployment Flow

1. **Build & Test** → Code is built and tested locally
2. **Deploy** → App code is deployed to Azure App Service
3. **Bootstrap SQL** → New step: SQL database user is created and configured
4. **Restart App** → New step: App Service is restarted to load new permissions

### Bootstrap Script Flow

```
GitHub Actions (Ubuntu)
  ↓
[1] Azure Login (using AZURE_CREDENTIALS secret)
  ↓
[2] Verify SQL database exists (az sql db show)
  ↓
[3] Read SQL initialization script
  ↓
[4] Substitute App Service name into SQL script
  ↓
[5] Execute SQL script using sqlcmd with Azure AD auth
  ↓
[6] Restart App Service to load new permissions
```

### SQL Operations

The `init-managed-identity.sql` script performs:

```sql
-- Create contained database user from the managed identity
CREATE USER [app-capdash-prod-prod01] FROM EXTERNAL PROVIDER;

-- Grant read permissions (SELECT on all tables)
ALTER ROLE db_datareader ADD MEMBER [app-capdash-prod-prod01];

-- Grant write permissions (INSERT/UPDATE/DELETE)
ALTER ROLE db_datawriter ADD MEMBER [app-capdash-prod-prod01];
```

## Environment Variables and Secrets

Required GitHub Actions secrets (set in repository settings):

| Secret | Purpose | Example |
|--------|---------|---------|
| `AZURE_SUBSCRIPTION_ID` | Azure subscription to deploy to | `844eabcc-dc96-453b-8d45-bef3d566f3f8` |
| `AZURE_RESOURCE_GROUP` | Resource group containing SQL Server | `rg-capdash-prod` |
| `AZURE_WEBAPP_NAME` | App Service name (determines SQL names) | `app-capdash-prod-prod01` |
| `AZURE_CREDENTIALS` | Service principal credentials (JSON) | Generated by `az ad sp create-for-rbac` |

The script automatically derives SQL server and database names from the AZURE_WEBAPP_NAME:
- `app-capdash-prod-prod01` → SQL server: `sql-capdash-prod-prod01`, Database: `sqldb-capdash-prod`

## Prerequisites

### Azure Side

1. **Service Principal** with permissions to:
   - Deploy to App Service
   - Query SQL database
   - Restart App Service

2. **SQL Database Azure AD Admin** configured
   - The service principal must have Azure AD admin role on SQL Server
   - This allows it to execute SQL commands using Azure AD authentication

3. **App Service Managed Identity** (System-Assigned)
   - Created automatically when App Service is deployed
   - Used for token-based authentication to SQL Database

### Deployment

The `AZURE_CREDENTIALS` secret should be a JSON object created by:

```bash
az ad sp create-for-rbac \
  --name "github-deployment-sp" \
  --role "Contributor" \
  --scopes /subscriptions/{subscription-id}/resourceGroups/{resource-group}
```

Output:
```json
{
  "clientId": "...",
  "clientSecret": "...",
  "subscriptionId": "...",
  "tenantId": "...",
  "activeDirectoryEndpointUrl": "...",
  "resourceManagerEndpointUrl": "...",
  "activeDirectoryGraphResourceId": "...",
  "sqlManagementEndpointUrl": "...",
  "galleryEndpointUrl": "...",
  "managementEndpointUrl": "..."
}
```

## Running the Bootstrap Manually

If needed, you can run the bootstrap script locally:

```powershell
./scripts/bootstrap-sql-managed-identity.ps1 `
  -AppServiceName "app-capdash-prod-prod01" `
  -SqlServerName "sql-capdash-prod-prod01" `
  -SqlDatabaseName "sqldb-capdash-prod" `
  -ResourceGroup "rg-capdash-prod" `
  -SubscriptionId "844eabcc-dc96-453b-8d45-bef3d566f3f8"
```

Prerequisites for local execution:
1. Azure CLI installed and authenticated: `az login`
2. PowerShell 7+ or PowerShell Core
3. sqlcmd utility (comes with SQL Server Management Studio or SQL Server tools)

## Troubleshooting

### "User already exists" error

This is normal and expected if the bootstrap is run multiple times. The SQL script includes idempotent checks and will skip creation if the user already exists.

### "Login failed for user" in app logs

If the app still fails after deployment:

1. Verify the app restarted successfully:
   ```bash
   az webapp show --resource-group rg-capdash-prod --name app-capdash-prod-prod01 --query state
   ```

2. Check Application Insights logs for detailed errors:
   - Azure Portal → App Service → Application Insights

3. Verify managed identity is created:
   - Azure Portal → App Service → Settings → Identity
   - Should show System-assigned: On

### sqlcmd: command not found

Install SQL Server tools:
- **Windows**: Install "SQL Server 2019 Command Line Utilities"
- **macOS**: `brew install mssql-tools`
- **Linux**: `apt-get install mssql-tools`
